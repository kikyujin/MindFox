## 1. MxBS とは何か

MxBS（Multitapps Extended Brain System）は、ゲーム NPC やシミュレーションエージェントに
「覚える・忘れる・思い出す」を与える記憶管理ミドルウェアである。

最大の特徴は **GM（ゲームマスター）が定義した因子空間** で記憶を管理すること。
自然言語の embedding モデルに頼らず、ゲームにとって意味のある軸だけで
記憶を検索・忘却・想起する。

```
記憶セル = テキスト + 16バイト因子ベクトル + アクセス制御 + 重要度
```

### 1.1 ポジショニング

```
┌─────────────────────────────────────────────┐
│  Game / Simulation (Unity / UE / Godot)     │
├─────────────────────────────────────────────┤
│  MxMindFox — マルチエージェント配線          │
├─────────────────────────────────────────────┤
│  MxBS — 因子ベクトル記憶管理                 │
│  SQLite のみ。外部依存ゼロ。                 │
└─────────────────────────────────────────────┘
```

MxBS は単体でも使える。MxMindFox はマルチエージェントのオーケストレーション層として
MxBS の上に構築される。

## 2. なぜ embedding を捨てたか

### 2.1 前世代の問題

MxBS の前身である xMBS Lite は、bge-m3（1024次元）による意味検索を採用していた。

| 問題 | 詳細 |
|---|---|
| **ベクトルサイズ** | f32 × 1024 = 4,096 bytes/cell。Nintendo Switch の空きメモリ内でのスケーラビリティに制約 |
| **外部依存** | ONNX Runtime + bge-m3 モデル（~400MB）+ sqlite-vec 拡張が必要 |
| **言語バイアス** | bge-m3 は日本語テキストに対する類似度判定が英語より弱い。「密会」と「秘密の交渉」が遠い等 |
| **ブラックボックス** | 「なぜこの検索結果が返ったか」を GM が理解・調整できない |
| **過剰な解像度** | 1024次元は「あらゆる自然言語の意味を弁別する」ために必要。ゲーム NPC の記憶検索にそこまでの弁別力は不要 |

### 2.2 核心的な問い

> ゲーム NPC の記憶検索に必要な「似てる」の基準を、
> 汎用の言語モデルに丸投げする必然性はあるか？

答えは **No** だった。

ゲーム NPC が「思い出す」必要があるのは、
「あの時怖かった」「あいつは信用できない」「金が絡んでた」
といった **ゲーム的に意味のある感触** であり、
原文の語彙的ニュアンスではない。
テキスト本体はセルに別途格納されているのだから、
検索のインデックスに自然言語の全次元は不要。

### 2.3 パラダイムシフト

従来の embedding は **言語空間の写像** を行っていた。
MxBS は **ゲーム空間への射影** を行う。

情報を「落としている」のではなく、
**最初から必要な軸だけに射影している**。
ロスではなく設計。

## 3. 因子ベクトル

### 3.1 仕様

```
型:     u8 × 16（固定長）
サイズ: 16 bytes
値域:   各軸 0〜255
検索:   コサイン類似度
```

16 バイト固定とした理由:

- **下限の合理性**: ゲーム用途で意味のある軸は実測で 4〜12 程度。16 は十分な余裕を持ちつつ、SIMD アライメントにも適合する
- **上限の合理性**: 実際のシナリオ（外交シム、RPG、カードゲーム）で 16 軸を全て意味ある軸で埋められることを Hawaii 2035 シナリオで実証済み
- **u8 の合理性**: ゲーム用途で「軍事関連度が 0.7823 か 0.7824 か」を区別する需要はない。0〜255 の 256 段階で十分

### 3.2 軸設計の原則 — 「重要な概念に複数軸を」

16 軸全てをユニークな概念で埋める必要はない。
**ゲームにとって重要な概念には、高い解像度を複数軸で与える。**

例: 外交シミュレーションで「外交」が核心なら:

| 軸 | 名前 | 意味 |
|---|---|---|
| 0 | pressure | 外交的圧力（対話的 ↔ 威圧的） |
| 1 | alignment | 勢力間協調（敵対 ↔ 同盟） |
| 2 | formality | 公式度（密室 ↔ 公式声明） |
| 3 | reciprocity | 互恵度（一方的要求 ↔ 対等な取引） |

「外交」に 4 軸割くことで、コサイン類似度が外交的な記憶同士を
自動的に高ランクに返す。軸の配分そのものがゲームデザインの宣言になる。

これは **D&D が STR/DEX/CON/INT/WIS/CHA の 6 軸を選んだのと同じ構造** である。
「この 6 つが冒険で重要だから」という設計判断が、
キャラクターの記憶検索にも適用される。

### 3.3 NA 軸の排除

使わない軸を 0 埋めや NA とする設計は採用しない。
代わりに「重要な概念に複数軸を」の原則で、16 軸全てを意味ある軸で埋める。
これにより追加のマスクベクトルが不要になり、16 バイト固定が維持される。

## 4. プリセットシステム

### 4.1 二層戦略

MxBS の因子空間は **プリセット** として JSON で定義される。

| フェーズ | 担当 | LLM | コスト |
|---|---|---|---|
| **デザインタイム** — プリセット設計 | GM + 高級 LLM | Claude Opus / GPT-4o / Gemini Pro | 1 回のみ |
| **ランタイム** — セルスコアリング | ゲームエンジン + 軽量 LLM | gemma4:e2b〜26b | セル生成ごと |

高級 LLM が必要なのは **「世界を設計する瞬間」だけ** であり、
世界が回り始めたらローカルの軽量 LLM で十分。
GM がシナリオを書くときは頭を使うが、
セッション中のダイスロールは機械的にやるのと同じ構造。

### 4.2 プリセット JSON フォーマット

```json
{
  "name": "hawaii_2035_diplomatic",
  "version": "1.0",
  "description": "外交シミュレーション用16因子プリセット",
  "axes": [
    {"index":  0, "name": "pressure",    "low": "対話的",    "high": "威圧的"},
    {"index":  1, "name": "alignment",   "low": "敵対",      "high": "同盟"},
    ...
    {"index": 15, "name": "power_shift", "low": "現状維持",  "high": "激変"}
  ]
}
```

### 4.3 プリセットはアセット商品になる

プリセット自体が **販売可能なデザインアセット** である。

- 「外交シム用」「RPG パーティ用」「カードゲーム用」「恋愛シム用」
- MULTITAPPS の本業であるデザインアセット販売と同じ事業構造
- Claude Opus や GPT-4o で設計された高品質プリセットに価値がある

## 5. セル構造

MindFox から継承したセル構造を、因子ベクトルに適合させる。

### 5.1 フィールド

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | INTEGER | セル ID（自動採番） |
| `turn` | INTEGER | 生成されたターン番号 |
| `agent_id` | TEXT | 所有エージェント |
| `text` | TEXT | 記憶のテキスト本体 |
| `features` | BLOB(16) | 因子ベクトル（u8 × 16） |
| `group_bits` | INTEGER | アクセス制御ビットマスク（最大 64 エージェント） |
| `mode` | INTEGER | UNIX パーミッション（例: `0o744` = 全員可読） |
| `price` | INTEGER | 重要度（0〜255）。高いほど忘れにくい |

### 5.2 SQLite スキーマ

```sql
CREATE TABLE cells (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    turn        INTEGER NOT NULL,
    agent_id    TEXT    NOT NULL,
    text        TEXT    NOT NULL,
    features    BLOB    NOT NULL,    -- 16 bytes exactly
    group_bits  INTEGER NOT NULL DEFAULT 0,
    mode        INTEGER NOT NULL DEFAULT 484,  -- 0o744
    price       INTEGER NOT NULL DEFAULT 100
);
```

sqlite-vec 拡張は **不要**。コサイン類似度はアプリケーション側で計算する。

## 6. 検索・忘却・アクセス制御

### 6.1 コサイン類似度

```
cos(a, b) = Σ(aᵢ × bᵢ) / (√Σaᵢ² × √Σbᵢ²)
```

u8 × 16 の dot product は 16 回の乗算 + 15 回の加算。
SIMD で数ナノ秒。数千セルの全スキャンでもマイクロ秒オーダー。

### 6.2 忘却

```
effective_score = cosine × (price / 255) × decay(turn_diff)
decay(Δt) = exp(-0.693 × Δt / half_life)
```

- `price` が高い記憶は長く残る（重要な出来事は忘れにくい）
- `half_life` ターンで記憶の重みが半減（デフォルト 8 ターン）
- `price = 255` は不滅セル（世界設定等）

### 6.3 アクセス制御

`group_bits` はエージェントごとのビットフラグ（1 エージェント = 1 ビット、最大 64）。
`mode` は UNIX パーミッション形式で owner / group / other の rwx を制御する。

```
セルが見える条件（mode ベース）:
  viewer が owner → mode の owner 部に r ビットがあるか
  viewer が group に含まれる → mode の group 部に r ビットがあるか
  それ以外 → mode の other 部に r ビットがあるか
```

- 密室で 2 人だけの交渉 → `mode=0o740`、2 人のビットだけ group_bits に ON
- 公式声明 → `mode=0o744`（other=r で全員可読）
- inner_voice（内なる声）→ `mode=0o700`（owner のみ）

### 6.4 Dream（将来拡張）

MindFox から概念を継承。「忘れかけた記憶の浮上」。
buried_score = price × (1 - decay) で「重要なのに忘れかけた」記憶を検出する。
実装はアプリケーション要件確定後。

## 7. 2 フェーズ設計

ゲームループとの統合を考慮し、記憶の生成とスコアリングを分離する。

```
Phase 1: ストーリー生成（ゲームループ内）
  agent1 行動 → テキストをスタック
  agent2 行動（agent1 のテキストがコンテキストに）→ スタック
  ...
  サマリー生成
  ※ Phase 1 中、記憶検索は前ターンまでの確定セルに対してのみ

Phase 2: 一括スコアリング（ターン末）
  全セクションを LLM でスコアリング → DB 保存
  ※ ゲームのロード画面中やターン間で実行可能
```

この分離により:

- ゲームループ中のレイテンシにスコアリングが影響しない
- スコアリングの LLM コールを将来的に並列化・バッチ化できる
- xMBS Lite の「store → index_pending」パターンと同じ思想

## 8. 実証結果

### 8.1 Hawaii 2035 シナリオ

| 項目 | 値 |
|---|---|
| シナリオ | 外交シミュレーション（ハワイ独立承認） |
| エージェント | 5 名（leader / stubborn / impulsive / mediator / contrarian） |
| ターン数 | 6 |
| 生成モデル | gemma4:26b（ローカル、Ollama） |
| スコアリングモデル | 同上 |
| プリセット設計 | Claude Opus 4（16 因子: 外交 4 / 内面 3 / パワー 3 / 情報 2 / 構造 4） |
| 総セル数 | 66（公開 + 私的 + サマリー） |
| スコアリング失敗 | **0 回** |
| 検索上限 | 5 件 / エージェント / ターン |
| ターン所要時間 | ~60 秒（生成 ~45s + スコアリング ~15s） |

### 8.2 確認された能力

- **因果関係の成立**: 5 件の記憶検索だけで、6 ターンにわたるキャラクターアーク（極秘接触→葛藤→決断→承認）が自然に生まれた
- **アクセス制御**: inner_voice（内なる声）は本人のみ参照可能。他エージェントは外面的な行動のみを記憶として受け取る
- **忘却**: 古いターンの記憶は decay により自然に順位が下がり、直近の重要イベントが優先される
- **言語非依存**: スコアリングの出力は u8 数値配列。日本語テキストでも検索精度に問題なし
- **∞ターン対応**: コンテキストサイズは検索上限（5 件）で一定。ターン数増加による破綻なし

### 8.3 数値比較

| | xMBS Lite（旧） | MxBS |
|---|---|---|
| ベクトルサイズ | 4,096 bytes | **16 bytes**（256 倍圧縮） |
| 外部依存 | SQLite + sqlite-vec + ONNX + bge-m3 | **SQLite のみ** |
| モデルファイル | ~400 MB | **0** |
| 検索の説明可能性 | ブラックボックス | **因子名で説明可能** |
| 言語依存 | あり | **なし** |
| 因子設計 | 不要（モデルが暗黙に決定） | **GM が明示的に設計** |

## 9. MindFox からの継承

MxBS / MxMindFox は MinaideFox → MindFox → MxMindFox の三世代目である。

### 9.1 継承する概念

| 概念 | 説明 |
|---|---|
| セル構造 | text + メタデータ + ベクトルの三位一体 |
| 忘却 | price × 時間減衰による自然な記憶の消失 |
| アクセス制御 | group_bits + mode による「誰が何を知っているか」 |
| Dream | 埋もれた重要記憶の浮上 |
| Pull 型 API | コールバックなし。ホストが能動的に取得 |
| ホスト駆動 | ターン順・LLM 選択・行動候補は全てホストが決定 |
| 全部セル | ワールド情報も性格もエージェント定義もセル |

### 9.2 捨てた概念

| 概念 | 理由 |
|---|---|
| Embedder trait (f32) | u8 × 16 の因子ベクトルに置換 |
| VecStore trait | sqlite-vec 不要。アプリ側 cosine で十分 |
| ONNX Runtime 依存 | embedding モデル自体が不要 |
| scene_id | MxMindFox 層で管理（MxBS は汎用） |
| 2 ステージ embedding (Pending → Indexed) | 2 フェーズ設計（Phase 1: 生成 → Phase 2: スコアリング）に進化 |
