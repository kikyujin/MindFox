## 1. MxChatterFox とは何か

MxChatterFox は、プレイヤーが **選択肢ではなく単語を選ぶ** ことで会話を進める会話エンジンである。

従来の ADV が「A / B / C から選択 → 分岐」という構造であるのに対し、
MxChatterFox は「単語を組み合わせる → テンプレートで会話文を生成 → NPC が因子ベクトルに基づいて応答する」という構造を取る。

```
従来 ADV:
  選択肢 A ─→ 固定分岐
  選択肢 B ─→ 固定分岐
  選択肢 C ─→ 固定分岐

MxChatterFox:
  [WHO] × [ACTION] × [WHERE/WHAT]
    → テンプレート展開 → プレイヤー発話文
    → NPC セリフDB を MxBS cosine 検索
    → NPC 発話文
    → （プレイヤーは何度でも続けられる）
```

### 1.1 核心原理: 全てが MxBS セル

MxChatterFox の最大の特徴は、**単語もセリフも NPC 性格も、全て MxBS の同一因子空間に存在する** ことである。

```
MxBS 因子空間（16次元）
  ├── 単語セル:  「魔女」「森」「どこ」... 各々に因子ベクトル
  ├── セリフセル: 「森の奥に住んでおるよ」... 各々に因子ベクトル
  └── NPC プリセット: 長老=mediator, openness=200 ... 因子ベクトル
```

検索は MxBS.search() の cosine 類似度。閾値分岐の if/else は不要。

### 1.2 設計方針

| 方針 | 説明 |
|---|---|
| **LLM 不使用（ランタイム）** | 全処理が MxBS cosine 検索 + テンプレート展開。虹コン Layer 2 と同思想 |
| **エンベディング不使用** | 16因子ベクトルの cosine で検索。外部エンベディングモデル不要 |
| **全て MxBS セル** | 単語・セリフ・NPC性格が同一因子空間。MxBS API だけで完結 |
| **キーワード＝アイテム** | 会話で入手した単語カードの因子ベクトルで、次の検索精度が上がる |
| **好きなだけ会話できる** | ターン制限なし。NPC のセリフDBが自然に枯れることで会話が終わる |
| **不完全情報でも進行可能** | 全キーワード収集は必須でない。情報の量と質でプレイヤーが自己判断 |
| **カスケード検索 + バックトラック** | 3語→2語→1語→汎用と自動フォールバック |

### 1.3 MindFox シリーズでの位置づけ

```
┌─────────────────────────────────────────────────────────┐
│  Game（Unity / Web / CLI）                               │
│  UI・ゲームループ・シーン管理                             │
├─────────────────────────────────────────────────────────┤
│  MxChatterFox            会話エンジン                     │
│  単語選択 → cosine カスケード検索 → セリフ選択            │
├───────────────┬─────────────────────────────────────────┤
│  MxMindFox    │  YamAMVA                                 │
│  Mood/判断補正 │  YAML シナリオディスパッチ                │
├───────────────┴─────────────────────────────────────────┤
│  MxBS                                                    │
│  記憶セル・因子ベクトル・忘却・ACL                        │
└─────────────────────────────────────────────────────────┘
```

### 1.4 虹コン Director Agent Layer 2 との対比

| 項目 | 虹コン Layer 2 | MxChatterFox |
|---|---|---|
| ユーザー入力 | 自然文メンション | **構造化された単語選択** |
| セリフ選択 | ChromaDB 近傍検索 or キーワードマッチ | **MxBS cosine カスケード検索** |
| ベクトル空間 | Gemini Embedding（高次元） | **MxBS 16因子（GM定義）** |
| 性格表現 | essential テンプレート（LLM に注入） | **因子ベクトル（セリフ自体に焼き込み）** |
| ランタイム LLM | 不使用（事前生成セリフ DB） | **不使用（セリフ = MxBS セル）** |
| 事前生成 | Grok reasoning でバッチ生成 | **bake_cerif + bake_16element パイプライン** |
| 同一返答回避 | top-K ランダム | **cosine 上位 K 件からランダム** |
| コスト | embedding 1 回 / メンション | **ゼロ** |

## 2. 会話フロー

### 2.1 基本サイクル

```
┌────────────────────────────────────────────────────┐
│  プレイヤー                                         │
│                                                     │
│  ① 単語スロットから単語を選ぶ                       │
│     [WHO: 魔女] [WHERE: 森] [ACTION: どこ]          │
│                                                     │
│  ② テンプレート展開 → プレイヤー発話文              │
│     「森に住むという魔女はどこにいるんだ？」         │
│                                                     │
├────────────────────────────────────────────────────┤
│  MxBS cosine カスケード検索                          │
│                                                     │
│  ③ 「魔女」の因子ベクトルで NPC セリフ DB を検索    │
│     → 候補 20件                                     │
│  ④ 候補を「森」の因子ベクトルで cosine 再フィルタ   │
│     → 候補 5件                                      │
│  ⑤ 候補を「どこ」の因子ベクトルで cosine 再フィルタ │
│     → 候補 1件                                      │
│                                                     │
├────────────────────────────────────────────────────┤
│  NPC 応答                                           │
│                                                     │
│  ⑥ ヒットしたセリフセルを返す                       │
│     → 「森の奥に住んでおるよ」                      │
│  ⑦ grants 処理: セリフに付随するキーワードを付与    │
│     → 🔑「森の奥」GET                              │
│                                                     │
├────────────────────────────────────────────────────┤
│  プレイヤーは続けて会話するか、離れるか選択          │
│  → 続ける場合 ① に戻る（ターン制限なし）           │
└────────────────────────────────────────────────────┘
```

### 2.2 バックトラック（自動フォールバック）

3語全てで絞り込めない場合、自動的に語数を減らして再検索する。

```
[魔女] [森] [どこ]  → cosine ヒットあり → そのセリフを返す
           ↓ ヒットなし
[魔女] [森]         → cosine ヒットあり → 「森のことを聞きたいのかの？」
           ↓ ヒットなし
[魔女]              → cosine ヒットあり → 「魔女のう……聞いたことはあるが」
           ↓ ヒットなし
汎用フォールバック   → 「んー、何を言ってるかわからんのう」
```

バックトラックは **単語を末尾から1個ずつ落とすだけ** のシンプルなアルゴリズム。

### 2.3 会話コンテキストスタック

会話中は **直近の話題対象** がセッションに保持される。

```
ターン 1: who=魔女 → コンテキスト: { topic_entity: "魔女" }
ターン 2: who=その人 → 「その人」= コンテキスト.topic_entity → 魔女に解決
ターン 3: who=その人, what=名前 → 魔女 + 名前 でカスケード検索
```

「その人」「それ」「さっきの」等の照応表現はセッション変数の参照で解決する。LLM による照応解析は不要。

### 2.4 連続会話の例

```
プレイヤー: [魔女] [探す] [このあたり]
  →「このあたりに住むという魔女を探しているんだが」
  → cosine: 魔女→探す→このあたり → ヒット
  NPC「ああ、それならこの先の森に住んでいるよ」
  → 🔑「森」GET（因子ベクトル付きで MxBS に登録）

プレイヤー: [その人] [年齢] [男女]
  →「その人物の年齢や性別について教えてくれないか」
  → cosine: 魔女→年齢 → ヒット / →男女 → 部分ヒット
  NPC「えらいべっぴんさんだと言うが……年齢はわからんね」
  → 🔑「べっぴん」GET

プレイヤー: [その人] [会いたい] [どこ]
  →「会いたいんだが、どこに行けばいい？」
  → cosine: 魔女→会いたい→どこ → ヒットなし
  → backtrack: 魔女→会いたい → ヒット（性格:陽気 のセリフ）
  NPC「いや、わからんね。恋人にでもしたいのかい？」
```

## 3. 単語スロット設計

### 3.1 スロット構造

プレイヤーに提示される単語は **スロット** に分類される。

| スロット | 役割 | 例 |
|---|---|---|
| WHO | 話題の対象（人物・モノ） | 魔女 / 鍛冶屋 / その人 / 盗賊 |
| ACTION | プレイヤーの意図 | 探す / 買う / 頼む / 聞く / 教えて |
| WHERE | 場所 | このあたり / 森の奥 / 街の外れ |
| WHAT | 属性・詳細 | 名前 / 年齢 / 男女 / 値段 / 危険 |

スロットの組み合わせは自由。全スロットを埋める必要はない（1〜3 スロットで会話成立）。

### 3.2 単語の入手方法

| 入手経路 | 説明 |
|---|---|
| 初期配布 | シーン開始時に基本単語セットを付与 |
| 会話中 GET | NPC の応答に含まれるキーワードが新規単語として追加（因子ベクトル付き） |
| アイテム連動 | ゲーム内アイテム入手に連動して単語が開放 |
| シーン遷移 | 新しいエリアに入ると場所・人物の単語が追加 |

### 3.3 単語 = MxBS セル

**単語はそれ自体が MxBS のセルである。** 因子ベクトルを持ち、セリフ検索のクエリとして機能する。

```
MxBS セル「森」:
  id: 1001
  owner: PLAYER_ID
  text: "森"
  features: [0, 180, 0, 0, 50, 0, 200, 0, 0, 0, 0, 0, 0, 0, 0, 0]
            ↑ 因子ベクトル: 「森」という単語の意味的位置
  price: 255 (immortal: 単語は忘却しない)
  meta: '{"type":"keyword","slot":"WHERE","source_npc":"village_elder","grants_turn":3}'
```

プレイヤーが「森」を選択すると、このセルの因子ベクトルがそのまま cosine 検索のクエリになる。

**単語の因子ベクトルは `bake_16element` で生成される。** セリフと同じ因子空間にマッピングされるため、cosine 類似度で自然に関連セリフがヒットする。

## 4. セリフ bake パイプライン

### 4.1 二段 bake

MxADVeng の baker.py と同じ「bake」概念。デザインタイムに全てを焼き込み、ランタイムは MxBS 検索のみ。

```
chara.md          world.md         archetypes.md
(NPC定義)         (世界設定)        (性格テンプレ)
    │                │                │
    └────────┬───────┘                │
             ▼                        │
       bake_cerif                     │
     (Claude / Grok バッチ)           │
             │                        │
             ▼                        │
        cerif.json                    │
  (ドメイン非依存セリフDB)            │
  (プレイヤー発話 + NPC応答ペア)      │
             │                        │
             └────────┬───────────────┘
                      ▼
              bake_16element
              (ゲーム固有の因子マッピング)
                      │
                      ▼
                MxBS プリセット
              (セリフセル + 単語セル)
```

### 4.2 chara.md — NPC 定義ファイル

NPC ごとに1ファイル。基本情報 + archetype + 知識を定義する。

```markdown
# 村の長老（Elder Gond）

## 基本情報
- 名前: ゴンド
- 職業: 村長（元漁師）
- 年齢: 68歳
- 性格アーキタイプ: mediator

## 知識

### 魔女エレナの居場所
- 情報: 森の奥に住んでいることを知っている
- 確度: 高（本人から聞いた）
- grants: [森の奥]

### 魔女エレナの外見
- 情報: 美人だという噂を聞いている
- 確度: 中（人づて）
- grants: [べっぴん]

### 魔女エレナの名前
- 情報: エレナという名前をうっすら覚えている
- 確度: 低（昔聞いた）
- requires: [べっぴん]
- grants: [エレナ]

### 魔女エレナの年齢
- 情報: 知らない
- 確度: なし
```

### 4.3 world.md — ワールド設定ファイル

ゲーム全体の設定。全 NPC で共有される背景情報のみを記述する。

**進行フラグ（キーワード所持による分岐・シーン遷移条件等）は MxYamAMVA 側で管理する（`mxyamva_spec.md` 参照）。**

```markdown
# ミラの森 — ワールド設定

## 主要人物
- 魔女エレナ: 森の奥に住む薬師。美貌で知られるが人嫌い
- 盗賊ギルド: 最近活発化。森の入り口付近に潜む

## 地理
- ミラの村: 交易路の要衝。長老ゴンドが治める
- 暗い森: 村の北。奥にエレナの小屋
- 北の山: 険しい山道。盗賊の隠れ家あり

## 時代背景
- 王国暦312年。辺境の村々で薬草の不作が続いている
- 王都から派遣された徴税官が各村を巡回中
```

### 4.4 bake_cerif — セリフ bake

bake_cerif は cerif.json を作る工程の総称。作り方は問わない。

| 方式 | 説明 |
|---|---|
| シナリオライター手書き | 少数NPC・こだわりのセリフ。品質最高 |
| Claude / Grok バッチ生成 | chara.md + world.md を入力して量産。レビュー必須 |
| ハイブリッド | ベースを自動生成 → ライターが味付け・加筆 |

いずれの場合も出力は同一フォーマットの cerif.json。

```json
{
  "npc_id": "village_elder",
  "archetype": "mediator",
  "cerif_db_id": "elder_phase1",
  "lines": [
    {
      "id": "elder_001",
      "player_text": "このあたりに住むという魔女を探しているんだが",
      "npc_text": "ああ、それならこの先の森に住んでいるよ",
      "topic_tags": ["魔女", "場所", "森", "探す"],
      "grants": ["森の奥"],
      "requires": []
    },
    {
      "id": "elder_002",
      "player_text": "その魔女はどんな見た目なんだ？歳はいくつぐらいだ？",
      "npc_text": "えらいべっぴんさんだと言うが……年齢はわからんね",
      "topic_tags": ["魔女", "外見", "年齢"],
      "grants": ["べっぴん"],
      "requires": []
    },
    {
      "id": "elder_003",
      "player_text": "そのべっぴんの魔女の名前を知らないか？",
      "npc_text": "あ、そういえば……エレナとか言ったかな",
      "topic_tags": ["魔女", "名前"],
      "grants": ["エレナ"],
      "requires": ["べっぴん"]
    },
    {
      "id": "elder_010",
      "player_text": "その魔女に会いたいんだが、どうすればいい？",
      "npc_text": "いや、わからんね。恋人にでもしたいのかい？",
      "topic_tags": ["魔女", "不明"],
      "grants": [],
      "requires": []
    },
    {
      "id": "elder_fallback",
      "player_text": null,
      "npc_text": "んー、何を言ってるかわからんのう",
      "topic_tags": [],
      "grants": [],
      "requires": []
    }
  ]
}
```

**プレイヤー発話がシーン/NPC ごとに変わる例:**

同じ [魔女][森][どこ] の単語選択でも:

| cerif_db | player_text | 状況 |
|---|---|---|
| `elder_phase1` | 「若い魔女はどの森にいるんだ？」 | 序盤、情報収集 |
| `hunter_phase2` | 「黒い森の魔女はどこに逃げたんだ」 | 終盤、追跡モード |

MxYamAMVA がどの `cerif_db_id` を使うかを制御するため、同じ NPC でもストーリー進行に応じてセリフセットが差し替わる。

**cerif.json はドメイン非依存。** 因子ベクトルはまだ含まれない。`topic_tags` が bake_16element の入力になる。

### 4.5 bake_16element — 因子マッピング

`bake_16element cerif.json preset.json` を実行すると、ゲーム固有の 16 因子プリセットに基づいて全セリフ・全単語に因子ベクトルを付与する。

```
入力:
  cerif.json       — セリフDB（topic_tags 付き）
  preset.json      — ゲーム固有の16因子定義

出力:
  MxBS プリセットデータ（セリフセル + 単語セル + NPC プリセット）
```

因子マッピングの方式:

| 方式 | 説明 |
|---|---|
| ルールベース | topic_tags → 因子スロットへの固定マッピング表で変換 |
| LLM スコアリング | gemma4:e2b〜26b で cerif.json の各行をスコアリング（デザインタイム） |
| ハイブリッド | ルールベースで粗付け → LLM で微調整 |

**bake_16element はデザインタイムのみ実行。** ランタイムでは MxBS にプリセット済みのセルを検索するだけ。

### 4.6 テンプレート量の見積もり

| カテゴリ | 数 / NPC | 備考 |
|---|---|---|
| 知識あり応答 | 30〜40 | 各知識項目 × バリエーション |
| 知識なし応答 | 20〜30 | archetype に応じた「知らない」バリエーション |
| リアクション・茶化し | 10〜20 | 性格による色付け |
| フォールバック | 3〜5 | 汎用「わからない」応答 |
| **合計** | **~100** | |

archetype ごとの共通パターンを `archetypes.md` から引くため、NPC 固有部分は知識定義のみ。

## 5. NPC 応答選択ロジック

### 5.1 cosine カスケード検索

**if/else の閾値分岐は使わない。** MxBS の cosine 類似度で自然に最適セリフが選ばれる。

```
プレイヤー入力: [word_1] [word_2] [word_3]
          │
          ▼
① コンテキスト解決
   「その人」→ session.topic_entity の因子ベクトルに置換
          │
          ▼
② requires チェック
   候補セリフの requires をプレイヤーのキーワード所持と照合
   → requires 未充足のセリフは候補から除外
          │
          ▼
③ カスケード cosine 検索（3語）
   word_1.features で NPC セリフ DB を MxBS.search()
     → 候補 N 件
   候補を word_2.features で cosine 再フィルタ
     → 候補 M 件
   候補を word_3.features で cosine 再フィルタ
     → 候補 K 件

   K ≥ 1 → ⑤ へ
   K = 0 → ④ バックトラック
          │
          ▼
④ バックトラック
   [word_1] [word_2] で再検索 → ヒットあり → ⑤ へ
   [word_1] のみで再検索       → ヒットあり → ⑤ へ
   全てヒットなし               → フォールバックセリフ → ⑤ へ
          │
          ▼
⑤ ランダム選択（同一返答回避）
   候補 K 件から 1 つをランダム選択
   直近 N ターンで使用済みセリフは除外
          │
          ▼
⑥ grants 返却
   セリフに grants フィールドがあれば、呼び出し元に返却する
   → 実際のインベントリ追加は MxYamAMVA の責務（MxChatterFox はステートレス）
          │
          ▼
⑦ 会話ログ記録（任意）
   会話内容を MxBS セルとして記録
   → NPC が「前に聞いたよね」と言える
```

### 5.2 cosine 検索の詳細

```python
def cascade_search(mxbs, npc_id, words, npc_lines_owner):
    """カスケード cosine 検索 + バックトラック"""

    # words = [word_cell_1, word_cell_2, word_cell_3]（各々 MxBS セル）
    for depth in range(len(words), 0, -1):
        # depth=3 → 3語全部, depth=2 → 2語, depth=1 → 1語
        query_words = words[:depth]

        # Step 1: 最初の単語で検索
        candidates = mxbs.search(
            query_features=query_words[0].features,
            searcher=npc_id,
            owner=npc_lines_owner,  # NPC のセリフセルだけを対象
            limit=20,
        )

        # Step 2: 残りの単語で順次フィルタ
        for word in query_words[1:]:
            candidates = [
                c for c in candidates
                if mxbs.cosine(c.features, word.features) >= COSINE_THRESHOLD
            ]

        # Step 3: requires チェック
        candidates = [
            c for c in candidates
            if player_has_all(c.meta.requires, player_keywords)
        ]

        if candidates:
            # 同一返答回避: 直近使用済みを除外
            fresh = [c for c in candidates if c.id not in recent_used]
            pick = random.choice(fresh if fresh else candidates)
            return pick

    # 全バックトラック失敗 → フォールバック
    return get_fallback_line(npc_id)
```

### 5.3 嘘つき NPC

嘘つき NPC（archetype の honesty 因子が低い）は、**嘘のセリフが正しい因子ベクトルを持つ** ことで実現する。

```
NPC「怪しい旅人」のセリフ DB:
  - text: "ああ、北の山にいるよ"（嘘）
    features: [魔女:高, 場所:高, ...]  ← 魔女+場所の cosine で正しくヒットする
    grants: ["北の山"]                  ← 嘘の情報がプレイヤーに渡る
    meta: {"lie": true}                ← ゲームロジック用フラグ
```

cosine 検索は「このセリフが嘘か真か」を判定しない。嘘セリフも正しい話題に対してヒットする。
プレイヤーは複数 NPC からの情報の矛盾に自分で気づく必要がある。

## 6. キーワード探索システム

### 6.1 キーワード＝アイテム＝MxBS セル

```
従来 ADV:    鍵を拾う → 扉に使う → 新エリア
ChatterFox:  単語を拾う → 会話に使う → 新情報
```

プレイヤーの所持キーワードは **因子ベクトル付きの MxBS セル** として管理される。
新しい単語を入手するとは、新しい因子ベクトルを手に入れることであり、
それによって **次の会話で cosine 検索の精度が上がる**（= より深い情報にアクセスできる）。

### 6.2 キーワード入手のフロー

```
MxChatterFox がセリフを返す: grants = ["べっぴん"]
    ↓
① MxYamAMVA が grants を受け取る
    ↓
② 「べっぴん」のセルが MxBS に存在するか確認
    → 存在する（bake_16element で事前登録済み）
    ↓
③ MxYamAMVA がプレイヤーの所持キーワードリストに「べっぴん」を追加
    → UI に新規カード表示（✨ NEW）
    ↓
④ 次の chatterfox 呼び出し時、available_words に「べっぴん」が含まれる
    → 「べっぴん」の因子ベクトルで新たな cosine 検索が可能に
```

**MxChatterFox 自体はキーワードの付与を行わない。** grants フィールドを返却するだけ。付与判断・インベントリ管理は MxYamAMVA の責務。

### 6.3 進行判定（不完全情報 OK）

特定のシーン遷移やイベントは、キーワードの **マッチ率** で判定する。

```json
{
  "event": "enter_forest",
  "required_keywords": ["森の奥", "魔女", "エレナ", "危険", "お守り"],
  "min_match_ratio": 0.6,
  "outcomes": {
    "1.0": "perfect_entry",
    "0.8": "good_entry",
    "0.6": "risky_entry",
    "below": "blocked"
  }
}
```

- 5/5 = 完全に準備万端 → 最良の展開
- 4/5 = 概ね把握 → 良い展開（1 つ見落とし）
- 3/5 = 最低限 → 進めるがリスクあり
- 2/5 = 情報不足 → 「もう少し調べてから来い」

## 7. MxBS 連携

### 7.1 因子空間の統合

MxChatterFox では、以下が全て同一の 16 因子空間に存在する。

| セル種別 | owner | price | 用途 |
|---|---|---|---|
| NPC セリフ | NPC_ID | 255 (immortal) | cosine 検索の対象 |
| 単語カード | PLAYER_ID | 255 (immortal) | cosine 検索のクエリ |
| 会話ログ | NPC_ID | 80〜150 | 「前に聞いたよね」判定 |
| NPC プリセット | NPC_ID | 255 (immortal) | archetype + 基本性格 |

```
プリセット例（RPG 聞き込みシナリオ）:
  [0]  topic_witch      : 魔女関連の話題度
  [1]  topic_location   : 場所・地理関連
  [2]  topic_person     : 人物属性（名前・年齢・外見）
  [3]  topic_danger     : 危険・警告関連
  [4]  topic_trade      : 商取引・物品関連
  [5]  action_search    : 「探す」系アクション
  [6]  action_ask       : 「聞く」系アクション
  [7]  action_buy       : 「買う」系アクション
  [8]  personality_open  : 外向性（archetype 由来）
  [9]  personality_honest: 誠実さ（archetype 由来）
  [10] personality_brave : 勇気（archetype 由来）
  [11] personality_suspicious: 警戒心（archetype 由来）
  [12..15]              : ゲーム固有の拡張用
```

### 7.2 会話ログ → MxBS 記録

会話内容を MxBS に記録することで、NPC が過去の会話を参照できる。

```
NPC が「前に聞いたよね」と言える:
  MxBS search(query_features=魔女.features, searcher=npc_id)
  → 「プレイヤーが魔女の場所を聞いた」ログセルがヒット
  → 「前にも言ったが、森の先だよ」テンプレート発火
```

### 7.3 MxMindFox Mood 連動（オプション）

MxMindFox の Mood を参照してセリフ選択に影響を与える。

- Mood: aggression 高 → cosine 候補のうち攻撃的なセリフが優先される
- Mood: friendliness 高 → 親切なセリフが優先される

実装: Mood の因子値をクエリベクトルに加算して cosine 検索の結果を偏らせる。

## 8. MxYamAMVA 連携

MxChatterFox のシナリオ統合は **MxYamAMVA**（YamAMVA の MxBS/MxMindFox/MxChatterFox 拡張）が担う。

詳細は `mxyamva_spec.md` を参照。以下は MxYamAMVA が管理する主要な責務:

| 責務 | 説明 |
|---|---|
| NPC 登場管理 | シーンごとにどの NPC が会話可能か |
| cerif_db 切り替え | ストーリー進行度で NPC のセリフ DB セット（`cerif_db_id`）を差し替え |
| キーワード分岐 | プレイヤーの所持キーワードによるシーン遷移判定 |
| chatterfox ノード | 固定セリフ区間（speak）↔ 自由会話区間（chatterfox）の切り替え |
| mxbs_push | シナリオイベントから MxBS への記憶書き込み |
| bake 管理 | chara.md + world.md → cerif.json → MxBS プリセットのパイプライン制御 |

MxChatterFox 自体は「渡された cerif_db_id の中で cosine 検索する」だけ。何を渡すかは MxYamAMVA が決める。

## 9. 障害耐性

| 障害 | 影響 | 対策 |
|---|---|---|
| MxBS 障害 | 全会話停止 | SQLite ファイルコピーからリストア |
| セリフセル欠損 | 特定話題の応答不能 | フォールバックセリフで代替 |
| 単語セル欠損 | 特定単語の検索不能 | UI から該当単語を非表示 |

外部 API 依存がゼロのため、ネットワーク障害の影響を一切受けない。

## 12. 用語集

| 用語 | 定義 |
|---|---|
| **単語スロット** | WHO / ACTION / WHERE / WHAT 等の分類枠。プレイヤーは各スロットから単語を選ぶ |
| **キーワード** | 会話で入手する単語カード。因子ベクトル付きの MxBS セル |
| **cerif.json** | bake_cerif が生成するドメイン非依存セリフDB。プレイヤー発話 + NPC応答のペアを含む |
| **cerif_db_id** | セリフDBセットの識別子。同一NPCでもストーリー進行で差し替わる |
| **bake_cerif** | cerif.json を作る工程の総称。手書き / LLM バッチ / ハイブリッドいずれも可 |
| **bake_16element** | cerif.json にゲーム固有の16因子ベクトルを付与するツール |
| **カスケード検索** | 複数単語の因子ベクトルで段階的に cosine フィルタする検索方式 |
| **バックトラック** | カスケード検索でヒットしない場合に単語数を減らして再検索する仕組み |
| **grants** | セリフに付随するキーワード付与指定。会話で新しい単語カードを得る |
| **requires** | セリフにアクセスするために必要なキーワード前提条件 |
| **会話コンテキスト** | セッション中の話題対象スタック。照応解決に使用 |
| **archetype** | NPC の基本性格パターン（stubborn / leader / mediator 等） |
| **player_text** | プレイヤー発話テンプレート。同じ単語選択でもNPC/シーンで文面が変わる |
